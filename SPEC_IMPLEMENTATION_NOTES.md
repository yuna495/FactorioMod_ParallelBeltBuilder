# Parallel Belt Builder 仕様実装メモ

本ドキュメントは、Parallel Belt Builderの実装における仕様上の不明点と、それを解決するために置いた仮定・前提条件を記録するものです。

---

## 第1段階の実装メモ

### 1. 展開方向について (第1段階時点)

* **不明点**: 第1段階は「4本固定」「追加3列をゴースト配置」とされていますが、左右どちらに展開するかが指定されていません。
* **仮定・設計**: 展開方向は「基準ベルトの進行方向の右側」に固定します。
  * 北向き (`defines.direction.north`): 右側（東方向、`+X`方向）に3列展開。
  * 東向き (`defines.direction.east`): 右側（南方向、`+Y`方向）に3列展開。
  * 南向き (`defines.direction.south`): 右側（西方向、`-X`方向）に3列展開。
  * 西向き (`defines.direction.west`): 右側（北方向、`-Y`方向）に3列展開。

### 2. 直線経路の判定について

* **不明点**: 「直線経路のみ」の判定条件。プレイヤーが一時的に蛇行した場合や、逆戻りした配置を行った場合の扱い。
* **仮定・設計**: 以下の条件をすべて満たすものを「直線経路」と判定します。
  * 記録されたすべてのベルトが同一の向き (`direction`) を持っていること。
  * 南北方向の場合は、すべてのベルトが同じX座標上に並んでいること。
  * 東西方向の場合は、すべてのベルトが同じY座標上に並んでいること。
  * 上記を満たさない場合は直線経路ではないと判断し、警告を表示して配置をキャンセルします。

### 3. 重複した配置の処理

* **不明点**: プレイヤーが同じタイルに重ねてベルトを置いた（または再配置した）場合の処理。
* **仮定・設計**: 同一座標への記録は重複排除（ユニーク化）を行い、1つ分のベルトとして処理します。これにより、同じ位置に複数のゴーストが重なって配置されるのを防ぎます。

### 4. 配置不能なタイルの定義とスキップ

* **不明点**: 「配置不能なタイルはスキップ」する具体的な挙動。
* **仮定・設計**:
  * `LuaSurface.can_place_entity` を使用して、その位置に実体の黄色ベルト (`transport-belt`) が配置可能かを検証します。
  * 水タイルの上や、他の衝突する実体エンティティが存在する場所では `can_place_entity` が `false` を返すため、その位置へのゴースト配置を自動的にスキップします。
  * すでに同種・同方向のベルトまたはゴーストが配置されている場合も、配置チェックによってスキップされます（余分な処理を行いません）。

### 5. 記録中の他エンティティの扱い

* **不明点**: 記録中にベルト以外のエンティティ（例：電柱、地下ベルト、インサータ）が置かれた場合の処理。
* **仮定・設計**: `on_built_entity` イベントで `name == "transport-belt"` (黄色ベルト) のみを選別して記録するため、他のエンティティは単純に無視されます。

## 第2段階の実装メモ

### 1. ベルト種類（黄・赤・青など）の自動取得

* **不明点**: 基準ベルトの種類を「自動取得」する条件とタイミング。
* **仮定・設計**:
  * 記録開始後に、プレイヤーが手動で配置した最初の `type == "transport-belt"` 型エンティティの名前（`entity.name`）を取得し、セッションデータに記録します。
  * これにより、バニラの黄（`transport-belt`）、赤（`fast-transport-belt`）、青（`express-transport-belt`）に加えて、他のModで追加された `transport-belt` 型の搬送ベルトも自動的に検出できます。
  * 同一セッション中に異なる種類のベルトが置かれた場合は、その配置を記録から除外すると同時に、プレイヤーへ警告文を表示します。

### 2. 展開方向（左右）の切り替え

* **不明点**: 進行方向に対する「左」または「右」の定義。
* **仮定・設計**:
  * 基準ベルトの前進方向ベクトルに対し、90度左または右へオフセットを計算します。
  * プレイヤーのGUIで `Left` が選ばれている場合は左側、`Right` が選ばれている場合は右側へ、指定本数（本数 - 1本分）の並行経路を展開します。

### 3. 通常配置（Normal）のアイテム消費と事前検証

* **不明点**: 通常配置時の「アイテム不足時は配置を中止し、部分配置は行わない」仕様の実装アプローチ。
* **仮定・設計**:
  * プレイヤーが `Confirm` をクリックした際、実際に配置されるべき有効なタイル（配置不能な水や障害物、既存の重複ベルトなどを除外した場所）の数を事前計算します（必要総数の算出）。
  * その必要総数と、プレイヤーのインベントリ内の同種ベルトの総数を比較します。
  * 所持数が不足している場合は、1枚も配置を行わずに処理を中止し、「必要数、アイテム名、所持数」を含む警告を表示します。これにより、部分的な配置によってベルト経路が途切れるのを防ぎます。
  * アイテムが足りている場合のみ、インベントリから必要数を一括で消費し、実体ベルトを配置します。

### 4. GUIの状態制御と位置保存

* **不明点**: プレイヤーごとのGUI状態管理と位置保存。
* **仮定・設計**:
  * 設定値（本数、展開方向、配置方法）は `storage.players[player_index]` に紐付けて管理し、セーブ・ロード後も維持します。
  * `on_gui_location_changed` イベントをハンドリングし、プレイヤーがGUIの表示位置を変更した場合はその座標を保存します。次回GUI作成時に保存された座標へ配置します。
  * 記録中（Recording）は、ゲームバランスや処理の破綻を防ぐため、本数、展開方向、配置方法のボタン操作を無効化（変更不可）します。記録停止後（Confirm待ち状態）は変更可能とし、Confirm時の配置に動的に反映させます。
